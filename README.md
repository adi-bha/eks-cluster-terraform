# Project: Building an EKS (Kubernetes) Cluster on AWS From Scratch

**Status:** In Progress
**Last Updated:** July 08, 2026

---

## 1. Problem Statement

If you want to run applications on Kubernetes in AWS, the easiest way to get started is a tool called `eksctl`. You run one command, and within about 15 minutes it creates an entire working Kubernetes cluster for you — networking, servers, security, permissions, everything.

That convenience is also the problem. `eksctl` hides *everything* it's doing behind that one command. It's like ordering a fully-built house from a catalog — you get to live in it, but you never learn how the plumbing, wiring, or foundation actually work. If something breaks, or if a company asks "why did you configure it this way," there's no real answer beyond "the tool did it."

The goal of this project is to open up that catalog house and build every single piece of it by hand, understanding what each piece does and why it's needed, using a tool called **Terraform** instead of `eksctl`.

---

## 2. What is Terraform, and why use it instead of eksctl?

Terraform is a tool that lets you describe infrastructure (servers, networks, permissions, etc.) as text files. Instead of clicking buttons in the AWS website or running one-off commands, you write down "I want a network that looks like X, a cluster that looks like Y," and Terraform figures out how to create it — in the right order, every time, reproducibly.

`eksctl` is purpose-built *only* for creating Kubernetes clusters. Terraform is general-purpose — most real companies use it to manage their entire cloud footprint (networks, databases, permissions, Kubernetes, everything) in one consistent way. Learning to build EKS via Terraform is directly transferable to how infrastructure is actually built and maintained in a professional environment, whereas `eksctl` is more of a "quick start" tool.

## 3. What are we actually building? (Plain-language tour)

Think of this project in layers, like building a house from the ground up:

1. **The land and roads (Networking / VPC)**
   Before you can build anything, you need a private, isolated network in AWS — like buying a plot of land and laying down roads. This includes public roads (for things that need internet access) and private roads (for things that should stay hidden from the internet).

2. **The permission system (IAM roles)**
   AWS needs to know who is "allowed" to do what. Before building the cluster, we create identity cards (IAM roles) for two things:
   - The Kubernetes control system itself
   - The worker computers that will actually run applications

3. **The control tower (EKS Control Plane)**
   This is the "brain" of Kubernetes — it decides where applications run, keeps track of health, and manages the overall system. AWS manages the hard parts of this for us, but we still need to tell it which network to live in and what security rules to follow.

4. **The trust bridge (OIDC Provider)**
   This is a slightly advanced but important concept: Kubernetes has its own internal identity system, completely separate from AWS's identity system. For applications running inside Kubernetes to be able to securely access AWS services (like reading a file from storage), AWS and Kubernetes need to be introduced to each other and learn to trust one another. This "introduction" is what the OIDC provider does.

5. **The workforce (Worker Nodes / Node Group)**
   These are the actual computers (EC2 instances) that will run your applications. The control tower assigns work to them, but they need to physically exist and be registered before anything can run.

6. **Individual permission badges (IRSA — IAM Roles for Service Accounts)**
   Once the trust bridge (step 4) exists, we can give *individual applications* running inside Kubernetes their own specific AWS permissions — like giving one employee a badge that only opens the supply closet, while another employee's badge only opens the server room. This is more secure than giving every application the same blanket permissions.

---

## 4. Why are we doing it in this specific order?

Each layer depends on the one before it — you can't build a second floor before the first floor exists. Specifically:

| Step | What we build | Why it must come at this point |
|------|----------------|-------------------------------|
| 1 | Network (VPC) | Nothing else can exist without a network to live in |
| 2 | Permission identities (IAM roles) | The cluster and worker nodes need identities before they're created |
| 3 | EKS Control Plane | Needs the network and its own identity to exist first |
| 4 | Trust Bridge (OIDC) | Can only be created *after* the control plane exists, because it needs information generated by the cluster itself |
| 5 | Worker Nodes | Need the control plane to exist so they have something to register with |
| 6 | Individual app permissions (IRSA) | Can only be created after the trust bridge (step 4) exists, since these permissions rely on that trust relationship |

We will build and verify each layer one at a time, rather than building everything at once, so that if something breaks, we know exactly which layer caused it — the same way you'd test plumbing before closing up the walls.

---

## 5. What does success look like?

By the end of this project:
- A working Kubernetes cluster will exist on AWS, built entirely from readable, reusable text files (no hidden magic).
- Applications running inside the cluster will be able to securely talk to other AWS services, each with their own narrowly scoped permissions — not one giant shared password.
- The entire setup can be destroyed and rebuilt identically at any time, which is a core principle of reliable, professional infrastructure management (this is called **reproducibility**).

## 6. Glossary (plain-language reference)

- **VPC (Virtual Private Cloud):** A private network inside AWS, isolated from the public internet unless explicitly allowed.
- **IAM (Identity and Access Management):** AWS's permission system — controls who/what can do what.
- **EKS (Elastic Kubernetes Service):** AWS's managed version of Kubernetes, a system for running and coordinating applications across many servers.
- **Control Plane:** The "brain" of Kubernetes that makes scheduling and management decisions.
- **Node / Node Group:** The actual servers that run your applications.
- **OIDC (OpenID Connect):** A standard way for two separate identity systems (in this case, Kubernetes and AWS) to trust each other.
- **IRSA (IAM Roles for Service Accounts):** A method for giving individual applications inside Kubernetes their own specific, limited AWS permissions.
